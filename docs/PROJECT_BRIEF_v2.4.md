*Brief version: 2.4 — all architectural decisions resolved*

## 0. Project Overview

### What it is
A personal CLI coding assistant built on a three-block routing architecture, designed around token efficiency, human-in-the-loop control, a structured documentation workflow, and full model configurability. It is not a design tool — it is a coding-first instrument.

### Problem it solves
**Cost efficiency.** Single-model alternatives such as Claude Code route every request through one expensive model regardless of task complexity. This system uses a router to match each request to the cheapest model capable of handling it.

**Documentation-integrated workflow.** A built-in three-tier documentation system (always-in-context project meta, on-demand detailed docs, session diary) allows the agent to maintain persistent context across sessions and projects in a structured, predictable way.

**Human-in-the-loop control.** Every file write, command execution, and external content fetch requires explicit user approval before execution. The agent observes freely; it modifies nothing without permission. On denial, the user can provide inline feedback and the worker regenerates — no work is wasted.

**Model flexibility.** Any model can be swapped by changing a single entry in `.agent.json`. No code changes required.

### Success criteria at 6 months
The system is in regular daily use by both authors for coding tasks. It runs stably on both macOS and Windows. The diary log has accumulated into a meaningful engineering record usable for retrospectives and interview preparation.

### Team and deployment model
Two authors. Each runs a fully independent instance of the system on their own machine. The application is self-contained, installed once globally, and invoked from any project directory. No shared state between instances. Mac ↔ Windows compatibility is a hard requirement.

### Core philosophy
1. **Token efficiency is a first-class concern.** Every architectural decision is evaluated against its token cost.
2. **Human is always in the loop.** The agent observes freely. It modifies nothing — files, commands, external fetches — without explicit user approval. Denial opens a correction loop, not an abort.
3. **Block 2 is an isolated experimental zone.** Its internal logic can be freely redesigned without touching Block 1 or Block 3.
4. **Files are shared memory between steps.** The filesystem is the source of truth. Context does not accumulate across Block 3 calls within a task.
5. **The working context is always the project root.** All file operations, command execution, and git operations are relative to the project root — the directory containing `.git/`, discovered by walking up from the invocation directory.

## 1. Three-Block Architecture

The system is divided into three functionally distinct blocks with stable contracts between them. The host Python process is always the orchestrating entity — models are called for specific scoped tasks, not given open-ended autonomy.

```
User Input
    │
    ▼
┌─────────────────────────────────────────────────┐
│  BLOCK 1 — Router                               │
│  Two sequential DeepSeek V4 Flash calls         │
│  Call 1: binary decision SIMPLE / COMPLEX       │
│  Call 2: file selection (+ instruction on SIMPLE│
└────────────────┬────────────────────────────────┘
                 │
        ┌────────┴────────┐
        │                 │
     SIMPLE            COMPLEX
        │                 │
        │         ┌───────▼──────────────────┐
        │         │  BLOCK 2                 │
        │         │  Orchestration System    │
        │         │  DeepSeek V4 Pro         │
        │         └───────┬──────────────────┘
        │                 │
        └────────┬────────┘
                 │
    ┌────────────▼──────────────────────┐
    │  BLOCK 3 — Worker                 │
    │  Qwen3 Coder Next                 │
    │  Executes structured tasks        │
    └───────────────────────────────────┘
```

### Block 1 — two-call design

Block 1 makes two independent, stateless Flash calls in sequence. Neither call shares conversation history with the other.

**Call 1 — routing decision:**
- Input: user request
- Output: `SIMPLE` or `COMPLEX`
- `temperature: 0`, `max_tokens: 10`

**Call 2 — file selection and instruction:**
- Input: user request + routing decision from Call 1 + full project file list + contents of all files in `docs/context/`
- The project file list is generated by the host calling `list_files()` directly before invoking Flash — this is a host-level call, not a model tool call. It respects `.agentignore` and produces a flat list of relative paths, regenerated fresh on every request.
- Output format depends on routing decision:
  - **COMPLEX:** bare JSON array of file paths — `["src/auth/register.py", "src/auth/validators.py"]`
  - **SIMPLE:** JSON object with file list and structured instruction:
    ```json
    {
      "files": ["src/auth/register.py", "src/auth/validators.py"],
      "instruction": {
        "task_description": "Add email validation to the registration form.",
        "constraints": "Use existing validator pattern from validators.py.",
        "expected_output": "Modified register.py with validation logic."
      }
    }
    ```
- `temperature: 0`, `max_tokens: 512`
- Two different system prompts for Call 2 depending on routing decision — same model, same call structure, branching output format.
- The host strips Markdown code fences and trims whitespace before calling `json.loads()`. Parse failure after stripping triggers one retry. If the retry fails, the host surfaces an error and halts.
- On SIMPLE path: if `docs/context/` is empty or absent, `constraints` in the instruction will be empty — expected behavior, not an error.

### Block contracts

**Block 1 → Block 3 (SIMPLE path):**
- User request (raw)
- File list + structured instruction from Call 2
- File contents (read by host from file list)
- Contents of all files in `docs/context/`
- Injected diary sections (sliding window)

**Block 1 → Block 2 (COMPLEX path):**
- User request (raw)
- File list (JSON array from Call 2)
- Contents of all files in `docs/context/`
- Injected diary sections (sliding window)

**Block 2 → Block 3 (per step, via host):**

The host calls DeepSeek Pro once per step during execution with the full planning conversation history plus the previous step's result object. Pro returns a five-section JSON instruction for that step:

```json
{
  "task_description": "Implement password hashing in the User model.",
  "files": ["models/user.py", "models/base.py"],
  "constraints": "Use bcrypt. Follow existing model conventions from base.py.",
  "expected_output": "models/user.py with User class and hash_password method."
}
```

The host reads the listed files from disk and assembles the full Block 3 prompt from this instruction. File contents (section 3) are always filled by the host — never by Block 2.

**Block 3 → host (per step result):**
```json
{
  "status": "success" | "partial" | "error",
  "files_written": ["path/to/file.py"],
  "commands_run": ["alembic upgrade head"],
  "denied": {
    "write_file": ["path/to/file.py"],
    "run_command": ["rm -rf dist/"],
    "read_file": ["path/to/secret.py"],
    "fetch_page": ["https://example.com"]
  },
  "errors": ["error message if any"]
}
```

Assembled by the host after each step completes — not model-generated. Empty keys are omitted. Subject to refinement during implementation.

**SIMPLE path:** Block 1 routes directly to Block 3. Host takes the file list and structured instruction from Call 2, reads file contents from disk, and assembles the five-section Block 3 prompt. Sections 1 (task description), 4 (constraints), and 5 (expected output) come from Flash's instruction object. Sections 2 (files provided) and 3 (file contents) are filled by the host. Block 2 is not invoked.

**COMPLEX path:** Block 1 passes to Block 2. Block 2 conducts a stateful planning session, produces a human-readable plan, and then on execution produces per-step JSON instructions via separate host calls. Full detail in Section 2.

## 2. Block 2 — Contract & v1 Implementation

Block 2 is not a single model call. It is a self-contained orchestration subsystem operating in two phases: planning and execution. Its input/output contract is stable and fixed. Its internal architecture is explicitly designed to be iterated on, replaced, or restructured without touching Block 1 or Block 3 — the description below is the v1 starting point, not a permanent specification.

### Input contract
- Routed request from Block 1
- Task type flag: `COMPLEX`
- Contents of all files in `docs/context/`
- Injected diary sections
- Initial file list from Block 1 Call 2

### Output contract
**Planning phase:** Human-readable plan shown to the user for approval.

**Execution phase (per step):** Five-section JSON instruction produced on demand — the host calls DeepSeek Pro once per step with the full planning conversation history plus the previous step's result object. Pro returns the JSON instruction. File contents are read by the host from disk after receiving the instruction — they are never part of Block 2's output.

### v1 Internal logic

**Phase 1 — Planning (stateful session)**

**Step 1 — Analysis.** DeepSeek Pro receives the user request and full context in a new planning conversation. It analyses the request and identifies what needs to be done. Uses Tier 1 tools to read additional files beyond the initial selection if needed. Scans all files in `docs/context/` for any `## Documentation` sections to determine which lowercase docs are available and pulls relevant ones via `read_file`.

**Step 2 — Decomposition.** Within the same planning conversation, Pro breaks the task into discrete steps. Each step must be scoped as a single coherent Block 3 call. Steps that affect the same file must be batched — no separate call per function within a file. Steps may include `run_command` execution where appropriate (e.g. running migrations, installing dependencies, running tests). The planning conversation history is maintained throughout and used in all subsequent phases.

**Step 3 — Plan approval.** The human-readable plan is shown to the user via `questionary`:

```
Plan (3 steps):
1. models/user.py — User model + password hashing
2. routes/auth.py — /login, /logout, /register endpoints
3. tests/test_auth.py — basic auth tests

Execute?
> Yes, proceed
  Modify plan
  Cancel
```

A soft warning is shown if the plan contains an unusually large number of steps, prompting the user to consider narrowing the task scope. The threshold is configurable in `.agent.json`. There is no hard step limit — the user is aware of the token cost and decides.

**Plan modification — conversation loop.** If the user selects "Modify plan", a non-streaming API call is made to DeepSeek Pro with the full planning conversation history plus the user's modification instruction. Pro regenerates the plan. The loop continues until the user selects "Yes, proceed" or "Cancel". All iterations accumulate in the planning conversation history.

```
❯ merge steps 1 and 2 into one call

Plan (2 steps):
1. models/user.py + routes/auth.py — User model, hashing, auth routes
2. tests/test_auth.py — basic auth tests

Execute?
> Yes, proceed
```

**Phase 2 — Execution (per-step orchestration)**

**Step 4 — Execution loop.** For each approved step:
1. Host calls DeepSeek Pro with: full planning conversation history + step number + previous step result object
2. Pro returns five-section JSON instruction for this step
3. If COMPLEX path and previous steps had denials: host prepends `## Previously denied operations` to the Block 3 prompt listing denied items from prior steps
4. Host reads files listed in the instruction from disk
5. Host assembles the Block 3 prompt from the JSON instruction and file contents
6. Calls Block 3 (Qwen3) with the assembled prompt
7. Block 3 generates output; host intercepts all Tier 2 tool calls
8. Correction loop runs if any Tier 2 operation is denied (see Section 4)
9. Host assembles the step result object
10. Rule-based step evaluation:
    - `success` → proceed to next step automatically
    - `partial` with only denials, no errors → proceed automatically, denied items included in next step's context via `## Previously denied operations`
    - `partial` with errors → halt, show user what failed, ask whether to proceed or abort
    - `error` → halt and notify user
11. If proceeding, step result feeds into next iteration — Pro receives it in context and adjusts the next step's instruction accordingly

**Files as shared memory.** Conversation history is NOT passed between Block 3 calls. Block 2 sees the step result object and reads the actual filesystem state via tool calls if needed. This prevents context accumulation and ensures Block 3 always works from the real current state.

## 3. Models & API Configuration

### Model roles

| Role | Model | Provider | Temperature | Max tokens |
|---|---|---|---|---|
| Router (Call 1) | DeepSeek V4 Flash | DeepSeek API | 0 | 10 |
| Router (Call 2) | DeepSeek V4 Flash | DeepSeek API | 0 | 512 |
| Orchestrator | DeepSeek V4 Pro | DeepSeek API | 0.6 | 8000 |
| Worker | Qwen3 Coder Next | OpenRouter → Parasail | 0.2 | 16000 |

**Note:** Both DeepSeek V4 Flash and V4 Pro are currently in preview status (released April 2026). Treated as stable for personal tooling — monitor for breaking changes. All model names live in `.agent.json`; a breaking API change requires one config edit.

### API keys
Two keys total:
- `DEEPSEEK_API_KEY` — covers both Flash (router) and Pro (orchestrator)
- `OPENROUTER_API_KEY` — covers Qwen3 Coder Next (worker)

Stored in `.env` at the project root. Never committed.

All model assignments, parameters, and system behaviour are defined in `.agent.json` at the project root. This file is the single source of truth for model and tool configuration.

Cost rates are set to `0.0` as placeholders. The user fills in actual rates from the provider's pricing page. Costs are always calculated from these config values — never from any API-returned cost field — ensuring cross-provider consistency.

### `call_llm()` abstraction
A single `call_llm(role, messages, **kwargs)` function resolves the provider and model from `.agent.json` and handles the API call. All blocks use this function exclusively — no hardcoded model names anywhere in the codebase.

`call_llm()` always returns token usage (input tokens, output tokens) in its response object regardless of whether tracing is enabled. The host tracks running session cost in-memory by multiplying token counts against the configured rates. Tracing writes to Postgres when available; cost display reads from the in-memory accumulator always.

### Message structure
Applies to all blocks:
- **System message:** Permanent context — all files in `docs/context/`, injected diary sections, agent behavioral rules.
- **User message:** Current task — the user request, file contents, task instruction, or planning input as appropriate to the block and call type.

This is a structural API contract, not a prompt engineering decision. The content of each message is defined per-block; this rule defines which role it goes in.

### Truncation detection
Every Block 3 response checks `finish_reason` in the API response. If `finish_reason == "length"`, the output was truncated at the token limit. A clear warning is surfaced to the user: output was cut — retry with a narrower task scope. Silent truncation is never acceptable. `max_tokens` for the worker is configurable up to `max_tokens_ceiling: 32000`.

### Model override
The active model for any role can be overridden at the CLI level or in-session without editing `.agent.json`:
- CLI flag: `agent --worker qwen/qwen3-coder-next`
- In-session command: `/model worker qwen/qwen3-coder-next`

Override applies for the current session only.

## 4. Tool System

### Tool tier definitions

Tools are divided into two tiers based on their effect on system state. Tier assignment is **role-dependent** — the same tool may have different tiers for different blocks.

**Tier 1 — auto-execute:** Tool call is executed immediately without user prompt. The operation is shown in a labelled panel in the UI. Used for read-only and low-risk information retrieval operations.

**Tier 2 — approval required:** Execution halts. Full details of the operation are shown to the user via a `questionary` prompt. User must explicitly approve before execution proceeds. On denial, a correction loop opens (see below).

### Tool assignments per block

| Tool | Block 2 | Block 3 |
|---|---|---|
| `read_file(path)` | Tier 1 | Tier 2 |
| `list_files()` | Tier 1 | — |
| `search_in_files(query)` | Tier 1 | — |
| `web_search(query)` | Tier 1, max 5 results | Tier 1, max 5 results |
| `fetch_page(url)` | Tier 1, truncated to 6000 chars | Tier 2 |
| `write_file(path, content)` | — | Tier 2 |
| `run_command(cmd)` | — | Tier 2 |

Block 3 has no access to `list_files` or `search_in_files` — file discovery is the responsibility of Block 1 and Block 2. Block 3's unexpected `read_file` call signals the planning phase missed something — it is surfaced as a Tier 2 pause.

Note: `list_files()` is also called directly by the host (not as a model tool call) during Block 1 file list generation. The tool tier system governs model-initiated tool calls only.

### Block 3 autonomy
Block 3 is not obliged to strictly follow its assigned scope. If Qwen3 notices something outside the task instructions — a missing import, a naming conflict, an inconsistency in adjacent code — it may propose and attempt its own corrections. This is intended behavior, not a violation of the architecture.

The Tier 2 permission system makes this safe regardless: any out-of-scope `write_file` call fires the standard approval prompt. The user sees exactly what Block 3 wants to touch and why, and decides. If approved, the write is recorded in the step result object and Block 2 sees the actual filesystem state on the next step.

Prompt engineering requirement: Block 3 must surface its observations in the streamed response text before attempting any out-of-scope write — explaining what it noticed and why it's proposing a change — so the user has context before the permission prompt appears.

### Tier 2 correction loop

On any Tier 2 denial in Block 3, a correction loop opens:

1. User selects "Deny — give feedback" and types a correction note
2. Host assembles the re-call with:
   - Original step instruction (including file contents frozen from the first Block 3 call — files are not re-read from disk during correction rounds)
   - `## Already executed in this step` section listing all operations approved so far in this step, so Block 3 does not re-propose them
   - Stripped previous output: natural language reasoning and generated code only — tool call JSON and tool results excluded
   - User's correction note
3. Block 3 is called with this assembled context
4. New output is shown, Tier 2 prompt reappears
5. Loop continues until the user selects "Allow" or "Skip step"

After each correction round, the running cost delta for that round is displayed inline so the user can gauge spend without running `/history`.

"Skip step" exits the loop immediately. The step is recorded as `partial` in the result object. Execution proceeds to the next step with the denied items noted in context. There is no cap on correction rounds — the user is aware of the token cost.

### Tier 2 prompt structure

```
╔══ Allow file write? ════════════════════════╗
║  src/auth/register.py  (+47 lines)          ║
║  > Allow                                    ║
║    Show diff                                ║
║    Deny — give feedback                     ║
║    Skip step                                ║
╚═════════════════════════════════════════════╝

╔══ Allow command? ═══════════════════════════╗
║  $ alembic upgrade head                     ║
║  > Allow                                    ║
║    Deny — give feedback                     ║
║    Skip step                                ║
╚═════════════════════════════════════════════╝

╔══ Allow fetch? ═════════════════════════════╗
║  https://docs.sqlalchemy.org/orm/...        ║
║  > Allow                                    ║
║    Deny — give feedback                     ║
║    Skip step                                ║
╚═════════════════════════════════════════════╝

╔══ Allow file read? ═════════════════════════╗
║  src/config/secrets.py                      ║
║  > Allow                                    ║
║    Deny — give feedback                     ║
║    Skip step                                ║
╚═════════════════════════════════════════════╝
```

"Show diff" appears only on `write_file` — not applicable to commands, fetches, or reads.
After `run_command` executes, stdout and stderr are streamed to the terminal in real time during execution inside a `rich` panel, and buffered simultaneously. The full buffer is returned to Block 3 as a tool result in the format `{"stdout": "...", "stderr": "...", "exit_code": 0}`.

### Stop mechanism
`Ctrl+C` at any point during active generation stops the stream immediately. Behavior is identical across Block 2 and Block 3:

1. Stream stops
2. Partial output is flushed to the terminal
3. Partial output is saved to context as a complete assistant message
4. Input prompt returns

The model sees its own partial output on the next call — it knows what it had already written and can course-correct precisely from that point. If the user gives feedback, the next call continues with full context of what was produced so far.

`Ctrl+C` during active generation is handled by a `KeyboardInterrupt` catch in the streaming loop. This is distinct from the session-level `SIGTERM`/`atexit` handler — catching `Ctrl+C` mid-stream does not trigger the full session close or diary generation flow.

`Ctrl+C` at the main input prompt (no active generation) triggers the interrupted session handler in Section 7.

### Tool implementations
All tools are implemented as lightweight Python functions. No external protocol, no Node.js dependency, no adapter layer. Tools are registered as OpenAI-compatible function definitions and passed to models via their function calling APIs. Pure stdlib where possible.

**`run_command` cross-platform and environment handling:**
The host determines the correct execution environment before running any command:
- **Primary:** if any file in `docs/context/` has a tech stack section specifying `uv`, commands are prefixed with `uv run` — automatically using the project's own `.venv`
- **Fallback (context absent or no `uv` mention):** if `uv.lock` exists at the project root, `uv run` prefix is applied
- **Otherwise:** command runs in the shell environment as-is — the user is responsible for having their project environment activated before invoking the agent

All commands execute in the project root directory regardless of the agent's installation location. For non-Python projects (Node, etc.) the host runs commands in the shell environment without modification. Windows Terminal and PowerShell are the supported Windows environments.

**`fetch_page` truncation:** content is truncated to `fetch_page_max_chars` (default 6000 characters, approximately 2000 tokens) before injection. Character-based truncation, consistent across platforms and model changes.

**`search_in_files` and `list_files`** respect `.agentignore` exclusion rules.

### `.agentignore`
File exclusion rules for `search_in_files` and `list_files`. Uses gitignore syntax. Lives at the project root. Generated by `/init` with sensible defaults:

```
# .agentignore
.git
.venv
venv
__pycache__
node_modules
dist
build
.next
*.pyc
*.lock
*.min.js
```

If no `.agentignore` is present, the agent falls back to the hardcoded default list above. The agent functions correctly in projects that have not been `/init`'d.

### Web search provider
Abstracted in `.agent.json` under `"web_search.provider"`. Supported values: `"brave"` (Brave Search API) or `"tavily"` (Tavily API). The corresponding API key is stored in `.env`. Switching providers requires one config change and one `.env` update — no code changes.

### Large file truncation
Files exceeding a size threshold are truncated before being passed to models: first `large_file_head_lines` lines + last `large_file_tail_lines` lines (defaults: 150 + 50), with an explicit note that content was truncated. Full file available on explicit request.

## 5. Documentation System

The documentation system is a built-in feature of the agent, active for every project. It operates on three tiers with distinct roles and access patterns.

### Directory structure

```
docs/
  context/          ← always injected at session start
    CAPITAL.md      ← generated by /init as starting point
  lowercase/        ← on-demand detailed docs
  diary/            ← session records, populated by /end
.agentignore        ← project root
.agent.json         ← project root
.env                ← project root
```

### Tier 1 — context/ (always in context)
All `.md` files in `docs/context/` are read at session start and injected into the system prompt for every model call. Small, high-signal, human-maintained. The user controls what is always-in-context by what they place in this directory.

The `/init` command generates `docs/context/CAPITAL.md` as the starting point. A project may have multiple context files as it grows: `CAPITAL.md`, `ARCHITECTURE.md`, `STACK.md`, etc. There is no naming convention requirement — directory membership is the signal.

**If `docs/context/` is empty or absent at session start:** a prominent but non-blocking warning is displayed — "No context files found in docs/context/ — project context limited. Run /init to create a template." The agent continues without project context. A flag is added to the system prompt indicating this so all models know they are operating without project-level guidance.

**`## Documentation` manifest.** Block 2 scans all files in `docs/context/` for any `## Documentation` sections. Listed files are the available lowercase docs. If no `## Documentation` section is found in any context file, no lowercase docs exist and Block 2 does not attempt to read any. This eliminates the missing-file edge case.

For small projects, a single `docs/context/CAPITAL.md` is sufficient. The system works correctly with context files alone.

### Tier 2 — lowercase/ (on-demand)
Detailed documentation: API references, implementation notes, library guides, component documentation. Not loaded by default.

Declared in any context file's `## Documentation` section. Block 2 reads these sections to determine which docs are available before deciding what to pull. Pulled via `read_file` (Tier 1 for Block 2) — no embedding or vector search.

Files live in `docs/lowercase/`. Naming is descriptive: `docs/lowercase/authentication.md`, `docs/lowercase/database-schema.md`.

### Tier 3 — diary/ (session summaries)
Auto-generated session records stored in `docs/diary/`. At session start, the last N entries are parsed and selected sections only are injected into the system prompt.

**Injected sections (from each entry in the sliding window):**
- `## Next session` — continuation pointer from the previous session
- `## Open questions` — unresolved risks and uncertainties
- `## Decisions` — established choices that should not be re-litigated

All other sections are for human retrospective use only and are never injected into context.

**Section header enforcement.** Diary entries are generated by Flash from the session trace. The generation prompt enforces exact markdown headers for all sections — no paraphrasing, no skipped sections, machine-parseable structure is a hard requirement. After generation, the host validates that all required section headers are present. If validation fails, Flash is re-prompted once. If the second attempt also fails, the entry is saved with a `[malformed]` marker and a warning is shown. The agent never crashes on a bad diary entry.

#### Diary entry format

```markdown
# YYYY-MM-DD — Session N

## State going in
What was working, what was broken, starting point for this session.

## Task
The specific task addressed in this session.

## Decisions
- Decision made — rationale.

## Alternatives rejected
- Option considered — why it was rejected.

## Files changed
- modified: path/to/file.py
- created: path/to/new_file.py

## What worked
Concrete wins from this session.

## What didn't / Mistakes
What failed, wrong assumptions, time lost.

## Open questions
Unresolved issues, uncertainties, risks identified.

## Next session
Where to pick up next time.

## Metrics
Token counts, cost, model calls, session duration.

## Notes
Free-form thoughts, context, observations.
```

#### Diary generation on `/end`
1. Host queries Postgres for all traces belonging to the current session
2. Host extracts a structured session summary from the trace — not raw JSONB dumps. Includes: user requests (Block 1 inputs), plans Block 2 produced (step titles and descriptions), step result objects (files written, commands run, denied items, errors), token and cost metrics from scalar columns. Full file contents and generated code payloads excluded — noise for diary purposes. Typical summary: 2–5k tokens regardless of session length.
3. DeepSeek Flash reads the structured summary and generates the diary entry
4. Host validates section headers; re-prompts once if malformed
5. Host determines diary entry number N: scans `docs/diary/` for existing files matching `YYYY-MM-DD-*.md` for the current date, finds the highest N, uses N+1. Handles gaps from manual deletions correctly.
6. If `docs/diary/` does not exist, host creates it automatically
7. Entry saved to `docs/diary/YYYY-MM-DD-N.md`
8. If the session had file writes: Flash generates a conventional-commit-style message and the agent branch is committed. If no writes occurred: commit is skipped, diary entry still generated.

The user does not write diary entries manually.

#### Diary unavailable (tracing disabled or Postgres unreachable)
If `--no-trace` is active or Postgres was unreachable at startup, `/end` skips diary generation and notifies the user. If the session had file writes, the agent branch is still committed with a generic message.

### Project initialisation — `/init`
Creates the following structure in the current directory:

```
docs/
  context/
    CAPITAL.md      ← template with required sections
  lowercase/        ← empty
  diary/            ← empty
.agentignore        ← default exclusions pre-populated
```

**Behavior when artifacts already exist:**
- `docs/context/CAPITAL.md` exists → skip, display warning: "CAPITAL.md already exists — skipping. Use --force to overwrite."
- `.agentignore` exists → skip with warning
- `docs/lowercase/`, `docs/diary/` → created only if absent, no warning
- `--force` flag overwrites existing files

The generated `docs/context/CAPITAL.md` template:

```markdown
# [Project Name]

## Purpose
What this project does and why it exists.

## Architecture
High-level structure and key design decisions.

## Tech stack
Languages, frameworks, key libraries, versions.

## Conventions
Naming conventions, code style, patterns to follow.

## Constraints
Hard limits, non-negotiable requirements, things to avoid.

## Documentation
List of available lowercase docs for this project:
- docs/lowercase/example.md — description
```

The `## Documentation` section is the manifest Block 2 uses to discover available docs. For projects with no lowercase docs, this section is left empty.

## 7. Version Control Integration

### Agent branch
At session start — before any model calls — the agent:
1. Detects the current git branch
2. Creates `agent/<current-branch-name>` if it does not exist, or resumes it if it does
3. Performs `git checkout agent/<current-branch-name>`

All file writes land on the agent branch. No commits are made to the user's named branch during a session. The user's working directory is on the agent branch for the duration of the session — they see both branches in the startup header and switch back manually after `/end`.

**No intermediate commits.** All approved writes accumulate in the working tree on the agent branch. A single batch commit is made at `/end`. In interrupted sessions, a single `[interrupted]` commit is made by the handler.

**Prerequisite check:** if the repository is in a detached HEAD state, the agent refuses to start, displays a clear error, and instructs the user to checkout a named branch first.

**Branch reuse:** if `agent/<current-branch-name>` already exists, it is reused. Commits accumulate across sessions on the same line of work.

**At `/end`:** if the session had file writes, all changes are committed. DeepSeek Flash generates a concise commit message from the session trace — conventional commit style, no mention of AI authorship. If no writes occurred, the commit is skipped. The user reviews, squashes, and merges into their branch at their discretion.

```
main
  └─ your-feature-branch             ← user merges here when ready
       └─ agent/your-feature-branch  ← agent works here
```

### Interrupted session handling
Both a `SIGTERM` handler and an `atexit` handler are registered at startup. On unexpected exit (terminal close, `SIGTERM`):

1. If there are uncommitted writes: agent branch is committed with `[interrupted]` prefix
2. Diary generation is skipped
3. Best-effort log entry written to Postgres if connection is still alive

`Ctrl+C` during active generation is handled by the streaming stop mechanism (Section 4) — it does not trigger this handler. `Ctrl+C` at the main input prompt triggers this handler.

If the branch commit fails, the branch is left dirty — recoverable manually via git. Best-effort only.

### No git repo fallback
If the project has no git repository, the agent warns clearly at startup and falls back to an in-memory undo stack for the session. The user is strongly encouraged to initialise git before any meaningful work.

### Suggested workflow per session
1. `agent` from any directory in the project
2. Agent detects current branch, creates or resumes `agent/<current-branch>`, checks out to it
3. Work on tasks sequentially — no intermediate commands needed between tasks
4. `/end` at day's end — diary generated, branch committed if writes occurred
5. User reviews diff, squashes commits, merges to their branch, switches back

## 8. Infrastructure

### Agent
The agent is installed globally via `uv tool install .` from the agent repository. After installation, the `agent` command is available from any directory without activation.

```bash
# Install once
uv tool install .

# Invoke from any project directory
agent
```

`uv tool install .` creates an isolated tool environment for the agent — separate from any project's environment. The agent's own dependencies never interfere with project dependencies. The `uv.lock` in the agent repository pins all agent dependencies.

### Working context
All file operations, command execution, and git operations performed by the agent are relative to the project root — the nearest ancestor directory containing `.git/`, discovered by walking up from the directory where `agent` is invoked. If no `.git/` is found within N levels, the agent falls back to `cwd()` with a warning.

The agent's own installation directory is irrelevant after startup. Every action the agent takes is in the context of the project it was invoked from.

### Config and `.env` discovery
On invocation, the agent walks parent directories until it finds `.git/`, treats that as the project root, and looks for `.agent.json` and `.env` there. Fallback to `cwd()` with a warning if no `.git/` found. Matches behavior of standard developer tools (`ruff`, `pyright`).

### Observability — custom Postgres
Self-hosted Postgres running as a single Docker container. Started once, runs in the background. The agent logs every model call to a `traces` table via `psycopg2`. Exact schema is an implementation-time decision — at minimum, columns cover: session ID, timestamp, block, role, model, input tokens, output tokens, latency, cost, input payload (JSONB), output payload (JSONB), error.

`docker-compose.yml` contains one service: Postgres. No other containers.

#### Postgres startup behavior
On agent startup, the host attempts to connect to Postgres with exponential backoff: 3 attempts at 1s → 2s → 4s intervals.

If all attempts fail:
```
⚠ Postgres unavailable — tracing disabled for this session.
  /end will not generate a diary entry.
  Start Postgres with: docker compose up -d
```
Agent continues normally — all core functionality remains available.

#### `--no-trace` flag
```bash
agent --no-trace
```
Also configurable in `.agent.json` under `"tracing.enabled": false`. When disabled: no Postgres connection attempted, `/end` skips diary generation, everything else works normally.

### Cross-platform
- All file path operations use `pathlib.Path`
- `run_command` detects OS, invokes the appropriate shell, handles project environment detection
- `rich` auto-detects terminal capabilities
- Windows Terminal and PowerShell are the supported Windows environments

## 10. Risks & Constraints

### Token overflow within a session
**Risk:** Context window fills during long sessions.
**Mitigations:**
- Warning displayed when context usage exceeds a configurable threshold
- Block 3 receives only files relevant to the current step
- Each call is scoped — no unbounded history accumulation

### Large file contents in context
**Risk:** Reading large files consumes significant tokens per step.
**Mitigation:** Files truncated to first 150 + last 50 lines with explicit notice. Configurable in `.agent.json`.

### Block 3 output truncation
**Risk:** Generated content exceeds `max_tokens: 16000` and is silently cut.
**Mitigation:** Every Block 3 response checks `finish_reason`. If `"length"`, clear warning shown. `max_tokens` configurable up to 32000.

### Correction loop token cost
**Risk:** Repeated denials burn tokens without progress.
**Mitigation:** Running cost delta displayed inline after each correction round. No artificial cap — user decides when to skip.

### DeepSeek V4 preview status
**Risk:** Both V4 Flash and V4 Pro are in preview. API behavior may change.
**Mitigation:** All model names in `.agent.json`. One config edit if a breaking change occurs.

### No git repository
**Risk:** File writes cannot be recovered via the agent branch mechanism.
**Mitigation:** Agent warns at startup, falls back to in-memory undo stack, strongly encourages git.

### Detached HEAD state
**Risk:** Agent cannot create a named branch.
**Mitigation:** Agent detects detached HEAD at startup, refuses to proceed, instructs checkout of a named branch.

### Postgres unavailable
**Risk:** Docker container not started, not ready, or Docker daemon not running.
**Mitigation:** 3 connection attempts with exponential backoff (1s → 2s → 4s). On failure: warn, disable tracing, continue. `--no-trace` bypasses Postgres entirely.

### `run_command` environment mismatch
**Risk:** Commands fail with "command not found" or missing dependencies.
**Mitigation:** Host detects project environment via context files tech stack (primary) or `uv.lock` (fallback). `uv run` prefix applied automatically for `uv` projects.

## 11. Prompt Breakdown (Implementation Order)

**Prompt 1 — Project skeleton**
`uv tool install .` deployment model. `call_llm(role, messages, **kwargs)` abstraction with always-returned token usage. In-memory session cost accumulator using config-defined rates. `.agent.json` config loader with full schema including cost rates. `.env` handling. `pathlib` throughout. Git root walk for project root discovery with `cwd()` fallback. Working context principle: all operations relative to project root. Postgres connection setup with `psycopg2` and retry logic.

**Prompt 2 — Custom tooling**
All tools as Python functions: `read_file`, `write_file`, `list_files`, `search_in_files`, `run_command`, `web_search`, `fetch_page`. OpenAI-compatible function schema definitions. Tool execution dispatcher with tier enforcement per block. `.agentignore` parser with hardcoded fallback defaults. `fetch_page` character truncation. `web_search` result capping. Provider abstraction for Brave/Tavily. Large file truncation logic. `run_command` environment detection: context files tech stack primary, `uv.lock` fallback, `uv run` prefix for `uv` projects. All commands executed in project root directory. `run_command` result format: `{"stdout": "...", "stderr": "...", "exit_code": 0}`. Real-time stdout/stderr streaming via `subprocess.Popen` with simultaneous buffering.

**Prompt 3 — Block 1: Router**
Host calls `list_files()` directly to generate project file list (not a model tool call). Two sequential Flash calls: Call 1 binary routing at `max_tokens: 10`, Call 2 at `max_tokens: 512` with routing-dependent output — bare JSON array on COMPLEX, JSON object with `files` + `instruction` on SIMPLE. Two distinct system prompts for Call 2. Host strips Markdown fences and trims whitespace before `json.loads()`. JSON validation with single retry on parse failure. Integration with main loop.

**Prompt 4 — Block 2 v1: Orchestrator**
Two-phase architecture. Planning phase: stateful DeepSeek Pro conversation, analysis and decomposition in maintained conversation history. Scan all context files for `## Documentation` sections to discover lowercase docs. Plan generation and display with soft large-plan warning. Plan approval via `questionary`. Plan modification loop: non-streaming calls with full `planning_messages` history, iterates until approval or cancel. Execution phase: per-step non-streaming Pro call with full planning history + step result → JSON instruction → host reads files → Block 3 prompt assembly. `## Previously denied operations` injection for COMPLEX path. Rule-based step evaluation. Tier 1 tool access only.

**Prompt 5 — Block 3: Worker**
Qwen3 Coder structured instruction handler. Full Tier 2 tool surface. `finish_reason` truncation detection with clear user warning. Tier 2 correction loop: denial opens feedback prompt, host assembles re-call with frozen original instruction + `## Already executed in this step` section + stripped previous assistant message + user note, loop until approval or skip. Inline cost display per correction round from in-memory accumulator. Step result object assembly by host. `KeyboardInterrupt` catch in streaming loop — partial output flushed to terminal, saved to context as complete assistant message, input prompt returns.

**Prompt 6 — Rich CLI & UX**
Streaming output. `arc` spinner. Tier 1 tool call panels. Tier 2 permission prompts with four options. `one-dark` syntax highlighting. Diff preview for file writes. `run_command` real-time stdout/stderr panel. ASCII startup header with models, directory, current branch, agent branch. `/history` token and cost display from in-memory accumulator. Truncation warning. Large-plan soft warning. Stop mechanism UX: `^C` message, partial output display. Step instruction spinner for COMPLEX execution phase.

**Prompt 7 — Session lifecycle**
`/init` project structure generation: `docs/context/CAPITAL.md` template with `## Documentation` section, `docs/lowercase/`, `docs/diary/`, `.agentignore`. Skip-with-warning behavior for existing files, `--force` override. `/end` flow: Postgres trace query → structured summary extraction → Flash diary generation → section header validation with single retry → `docs/diary/` auto-creation → N determination by scanning existing entries → diary file write → conditional Flash commit message and agent branch commit (skipped if no writes). `/model` override. Context files and diary section injection at session start. Missing context warning and absent flag. Git branch detection, agent branch creation/resume, explicit `git checkout` at startup before model calls. Detached HEAD check. No-git fallback with warning. `SIGTERM`/`atexit` handler for interrupted session (distinct from `KeyboardInterrupt` streaming stop).

**Prompt 8 — Observability**
Postgres `traces` table setup via `psycopg2` (schema decided at implementation time). Logging wrapper around `call_llm()` — every model call recorded automatically. Session ID management. `--no-trace` flag and `tracing.enabled` config handling. Graceful degradation when Postgres unreachable. In-memory token/cost accumulation always active independent of tracing state.
